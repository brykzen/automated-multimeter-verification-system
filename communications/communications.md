# Communication Architecture — Interface Protocols and Data Flow

## Overview

This document defines the communication framework of the automated multimeter verification system. The architecture is designed to ensure synchronous instrument control, real-time hardware switching, and high-integrity data persistence across heterogeneous platforms.

The system integrates four primary communication channels:
* **Instrument Control Layer:** GPIB (IEEE-488.2) via NI-VISA.
* **Hardware Interfacing:** SPI over Serial (USB) via Arduino (LINX).
* **Data Persistence:** ODBC (PostgreSQL over LAN).
* **Reporting Engine:** CLI-based IPC (LabVIEW to Python Subprocess).

## 1. Instrumentation Bus — GPIB (IEEE-488)

### Protocol Specifications
All metrological instruments are interfaced via the **GPIB (General Purpose Interface Bus)**, utilizing the **NI-VISA** (Virtual Instrument Software Architecture) library as the abstraction layer.

* **Controller:** PC with NI-GPIB interface.
* **Addressing:** Unique primary addresses (PAD) assigned to each device (1–30).
* **Standard:** IEEE-488.2 compliant command sets (SCPI).

### Instrument Inventory
| Instrument | Model | PAD | Protocol Role |
| :--- | :--- | :---: | :--- |
| **AC Power Source** | HP 6812A | 1 | Stimulus Control |
| **DC Power Source** | HP 6030A | 2 | Stimulus Control |
| **Electronic Load** | HP 6060B | 3 | Load Modulation |
| **Reference Standard** | HP 34401A | 4 | Measurement Acquisition |

### Multimeter Abstraction Layer (HAL)
A custom Hardware Abstraction Layer (HAL) was developed in LabVIEW to standardize communication regardless of the multimeter model. This enables "Plug-and-Play" support for new instrumentation by implementing a common API:
* `Initialize`: Session opening and identity (IDN?) check.
* `Configure`: Setting function (V/A/Ω) and range.
* `Fetch`: High-resolution data acquisition.

## 2. Hardware Control — SPI via Arduino (LINX)

### Physical Interface
The **Arduino Nano** acts as a bridge between the USB serial port and the high-speed **SPI (Serial Peripheral Interface)** bus used by the relay matrix.

* **Interface:** USB-Serial (Virtual COM Port).
* **Middleware:** NI LabVIEW LINX Toolkit.
* **Bus Speed:** Optimized for millisecond-range relay actuation.

### Logic Execution
1. LabVIEW computes a **16-bit control word** representing the desired relay state.
2. The word is transmitted to the Arduino via the LINX protocol.
3. The Arduino shifts the data to the **74HC595 registers** via SPI.
4. Latched outputs drive the power MOSFETs for physical signal routing.

## 3. Data Persistence — ODBC and PostgreSQL

### Database Connectivity
The system utilizes the **NI Database Connectivity Toolkit** to perform SQL operations on a **Raspberry Pi 5** server over the local area network (LAN).

* **Driver:** PostgreSQL ODBC 32-bit Driver (System DSN).
* **Authentication:** SCRAM-SHA-256 encrypted handshake.
* **Network Access:** Controlled via `pg_hba.conf` for specific workstation IPs.

### Connection Workflow
**LabVIEW** → **ODBC Manager** → **LAN (TCP/IP)** → **PostgreSQL (Port 5432)**

---

## 4. Reporting Interface — CLI Subprocess (Python)

### Automated Batch Processing
For the generation of technical certificates, LabVIEW utilizes an **Inter-Process Communication (IPC)** strategy based on system-level calls (`System Exec.vi`).

1. **Trigger:** The operator selects a batch of verified multimeters in the LabVIEW UI.
2. **Command:** LabVIEW executes a Python script via the Windows Command Line (CMD).
3. **Arguments:** The script receives a list of unique Verification IDs as input parameters.
4. **Execution:**
   * Python connects to PostgreSQL via `psycopg2`.
   * Data is processed and formatted using the `ReportLab` library.
   * **Massive Printing:** The script generates and saves multiple PDF reports in a single execution cycle.
5. **Feedback:** The script updates the `report_flag` in the database upon successful generation.

```bash
# Example of CLI call generated by LabVIEW
python report_generator.py --ids 124 125 126 --output_path "C:/Reports/"
```

## Connectivity Troubleshooting

This section provides a standardized diagnostic protocol for the most common communication interface failures.

### Instrumentation (GPIB)
* **Runtime Verification:** Ensure the **NI-VISA Runtime** engine is active and the GPIB interface is recognized in NI MAX (Measurement & Automation Explorer).
* **Physical Layer:** Verify that the total cable length does not exceed 20 meters and that the **star topology** limits are respected to avoid signal reflections and timing errors.
* **Addressing:** Confirm that there are no PAD (Primary Address) conflicts between instruments on the bus.

### Hardware Control (Arduino / SPI)
* **Port Mapping:** Validate the correct COM port assignment within the `configuracion.ini` file.
* **Resource Locking:** Ensure that no other serial monitors (e.g., Arduino IDE Serial Monitor) or third-party applications are locking the COM port.
* **Firmware Sync:** Verify that the Arduino Nano is correctly flashed with the compatible LINX firmware for the current LabVIEW version.

### Database and Network (LAN)
* **Connectivity Test:** Perform an **ICMP echo request (ping)** to the Raspberry Pi 5 IP address to verify network reachability.
* **Driver Architecture:** Confirm that the **32-bit PostgreSQL ODBC DSN** is correctly configured via `odbcad32.exe` (located in `SysWOW64`), as LabVIEW 32-bit cannot interface with 64-bit DSNs.
* **Server Status:** Ensure the PostgreSQL service is active and the `pg_hba.conf` allows the specific workstation's IP range.

## Summary

The communication architecture provides a layered and modular approach to system integration. By leveraging **industry-standard protocols (GPIB, SPI, ODBC)** and **OS-level scripting (Python)**, the platform ensures high reliability in instrument control and seamless data flow. 

The implementation of a CLI-based IPC for massive reporting and a custom HAL for instrumentation allows for independent upgrades of each communication layer without compromising overall system integrity, making it a robust solution for high-precision industrial verification environments.
